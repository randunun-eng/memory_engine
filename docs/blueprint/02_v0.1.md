# Wiki v3 — The Neural Memory Engine (v0.1)

> A generic, open-source memory orchestrator for agentic systems.
> Biologically-inspired. Crash-resistant. Privacy-first. Self-healing.
> Builds on Karpathy's LLM Wiki (v1) and rohitg00's LLM Wiki v2.

## Revision notes — what changed from v0.0

This revision addresses six design contradictions identified in post-hoc review. Scope is intentionally narrow: fix the contradictions, do not expand features. All changes listed here; the rest of the document is unchanged from v0.0 except where fixes touch it.

| # | Problem in v0.0 | Resolution in v0.1 | Sections touched |
|---|---|---|---|
| 1 | Event immutability contradicted PII rescrubbing | Events are strictly immutable. Rescrubbing affects only derived state. Leaked raw events trigger protective forgetting, not edits. | §4.1, §4.2, §4.8, §16 |
| 2 | Content-hash UNIQUE contradicted reinforcement-via-repeat-observation | Split into two concepts: `idempotency_key` (unique, caller-supplied) for exactly-once; `content_hash` (non-unique) for reinforcement and tombstone matching. | §4.1, §4.2, §5.1 |
| 3 | Retrieval wrote on the hot path (blocks read replicas, serializes under load) | Retrieval is pure-read. It emits a `retrieval_trace` event. The consolidator applies LTP updates in batch from those events. | §4.7, §5.2 |
| 4 | "In-place update" during reconsolidation violated event-sourcing | Every neuron mutation emits an event (`neuron_update`, `neuron_supersede`, `neuron_reinforce`). "In-place" was the wrong word; replay reproduces all state by replaying these events in order. | §5.5, §16 |
| 5 | `scope_leak_impossible` invariant referenced a `retrievals` table that did not exist | Added `retrievals` derived table populated by consolidator from `retrieval_trace` events, bounded to a rolling window. | §4.5, §4.8 |
| 6 | Scope-raising rule was written as a self-interrupting sentence | Rewritten cleanly. Scope can be tightened (raised toward `private`) automatically by content signals; loosening requires an explicit re-classification event. | §6.2 |

Also folded in as collateral fixes, because they are required by the above:

- Replaced `libsodium sealed_box` with `libsodium secretbox` (sealed boxes are asymmetric; we need symmetric authenticated encryption at rest).
- Added `embedder_rev` column on `neurons` (promised in v0.0 §7 but missing from the schema).
- Added an explicit concurrency model for neuron updates (required by fix #4).

Nothing else was changed. The 1500-odd other lines of v0.0 still apply.

## 0. Preamble

Most memory layers for LLM agents fail in the same three ways:

1. **They accumulate forever.** Every observation becomes a permanent fact. No decay, no supersession, no pruning. The knowledge base becomes a junk drawer within weeks.
2. **They crash under their own weight.** Multiple services on multiple ports, in-memory state lost on restart, no transactional boundaries, no circuit breakers. Operators learn to restart instead of fix.
3. **They leak.** PII, secrets, and private thoughts flow straight from ingestion into embeddings and cloud LLMs. Privacy is bolted on after the first incident.

Wiki v3 is designed to not do any of that. It takes one metaphor seriously — the human brain — and lets that metaphor constrain the design. Neurons fire, synapses strengthen, unused paths fade, contradictions trigger reconsolidation, sleep consolidates, and a blood-brain barrier keeps private things private.

The metaphor is a source of vocabulary and intuition, not a correctness criterion. Where it stops helping, drop it.

## 4. Component Catalog

### 4.1 Sensory Layer (ingress)

**Purpose.** Receive raw input, turn it into a normalized, classified, scrubbed `Event`, and emit exactly one row to the event log per accepted input.

**Submodules.** Adapters, Normalizer, Classifier, PII Scrubber — as in v0.0.

**Dedup and idempotency (revised in v0.1).** The Sensory Layer resolves two distinct questions that v0.0 conflated:

- *Is this the same delivery as a retry?* Answered by the caller-supplied `idempotency_key`. If present and seen within the retention window, the existing event_id is returned with no new row appended.
- *Is this the same content as before?* Answered by `content_hash`. Same content is **appended as a new event**. Repetition is a valid reinforcement signal for the consolidator. The only time a matching `content_hash` is rejected is when it appears in the `tombstones` table (protective forgetting).

Flow:

```
raw input → normalize → classify → scrub → compute content_hash
  │
  ├─ if idempotency_key already seen → return existing event_id, stop
  ├─ if content_hash in tombstones    → reject, do not append
  └─ else                              → append event, forward to working memory
```

**Rescrubbing (clarified in v0.1).** The Sensory Layer tags each event with the `scrubber_rev` in effect at write time. When the scrubber ruleset is updated, **events themselves are not modified**. See §4.2 and §4.8 for how the system reacts to scrubber-rule updates.

**Failure modes.** Malformed input → reject + log, no append. Duplicate idempotency key → return original event_id. Tombstoned content → reject + log. Scrubber miss (discovered later) → see §4.8 healing path.

**Connections.** Emits to Event Log only. Never touches neurons directly.

### 4.2 Event Log (WAL)

**Purpose.** The only source of truth. Append-only, content-hashed, monotonically ordered, durable.

**Revised schema.**

```sql
events(
  id              BIGSERIAL PRIMARY KEY,
  ts              TIMESTAMPTZ NOT NULL DEFAULT now(),
  type            TEXT NOT NULL,
  scope           TEXT NOT NULL,
  source          TEXT NOT NULL,
  payload         JSONB NOT NULL,

  content_hash    BYTEA NOT NULL,       -- NOT UNIQUE (v0.1); repeats are reinforcement signals
  idempotency_key TEXT,                 -- nullable; caller-supplied
  scrubber_rev    INT NOT NULL,
  parent_event_id BIGINT REFERENCES events(id),

  UNIQUE (idempotency_key)              -- partial index: WHERE idempotency_key IS NOT NULL
);

CREATE INDEX ON events (content_hash);
CREATE INDEX ON events (type, ts DESC);
```

**Rules (reinforced in v0.1).**

- Events are **truly immutable** once appended. No field may be updated. No exception for rescrubbing.
- Events may only leave the table via **protective forgetting**, which deletes the row and tombstones the `content_hash`.
- All other tables (neurons, synapses, episodes, patterns, retrievals) are rebuildable from events.
- Event types added in v0.1: `retrieval_trace`, `neuron_reinforce`, `neuron_update`, `neuron_supersede`. See §4.7 and §5.5.

**What rescrubbing means in v0.1.** When the PII ruleset advances from `scrubber_rev = 3` to `scrubber_rev = 4`, the healer does **not** touch events. It re-derives affected neurons under the newer rules and supersedes the old ones. The events still record what was actually ingested; this is the historical record and should be precious. If a specific event itself contains a live secret that was missed, that is a security incident: the fix is protective forgetting of that event, not editing.

**Trade-off acknowledged.** This means raw unscrubbed PII may sit in the event log between ingestion and detection. That window is the honest risk of this design. The options were (a) mutate events on rescrub, which breaks replay and audit, or (b) accept a detection window and fix via hard-delete when discovered. We chose (b). Deployments holding extremely sensitive data should tighten scrubber rules conservatively before ingest, not loosely and rely on rescrub.

### 4.4 Privacy Layer (Blood-Brain Barrier)

Unchanged from v0.0 **except** the Vault section:

- **Secrets Vault.** Encrypted store using **libsodium `secretbox`** (symmetric, authenticated) with a master key read from env, keyring, or HSM at startup. (v0.0 said "sealed box," which is asymmetric and the wrong primitive for at-rest self-access.) Secrets in neurons are replaced by opaque references like `vault://abc123`. Retrieval returns the reference only; unseal is a separate authenticated call.

### 4.5 Storage (the Cortex)

**Revised neuron schema (v0.1 adds `version` and `embedder_rev`, adds the `retrievals` table).**

```sql
neurons(
  id              BIGSERIAL PRIMARY KEY,
  kind            TEXT NOT NULL,
  tier            TEXT NOT NULL,
  content         TEXT NOT NULL,
  embedding       VECTOR(N),
  embedder_rev    INT NOT NULL,                    -- v0.1: tracks which embedder produced the vector
  activation      REAL NOT NULL DEFAULT 0.5,
  confidence      REAL NOT NULL DEFAULT 0.5,
  scope           TEXT NOT NULL,
  created_at      TIMESTAMPTZ NOT NULL,
  last_fired_at   TIMESTAMPTZ NOT NULL,
  fire_count      INT NOT NULL DEFAULT 0,
  source_count    INT NOT NULL DEFAULT 1,
  superseded_by   BIGINT REFERENCES neurons(id),
  superseded_at   TIMESTAMPTZ,
  provenance      JSONB NOT NULL,
  version         INT NOT NULL DEFAULT 0           -- v0.1: optimistic concurrency token
);

synapses(...)            -- unchanged from v0.0
episodes(...)            -- unchanged from v0.0
patterns(...)            -- unchanged from v0.0
vault(...)               -- unchanged from v0.0
tombstones(...)          -- unchanged from v0.0
healing_log(...)         -- unchanged from v0.0

-- NEW in v0.1: derived table for retrieval invariants
retrievals(
  id                BIGSERIAL PRIMARY KEY,
  request_id        TEXT NOT NULL,
  ts                TIMESTAMPTZ NOT NULL,
  requestor_scope   TEXT NOT NULL,
  returned_neuron_ids  BIGINT[] NOT NULL,
  returned_max_scope   TEXT NOT NULL,
  degraded_streams  TEXT[]
);
CREATE INDEX ON retrievals (ts DESC);
-- Bounded: healer prunes rows older than retrieval_trace_retention (default 7 days).
```

The `retrievals` table is a derived rolling log, populated by the consolidator from `retrieval_trace` events in the main event log. It exists for invariant checks and operational visibility. Like all derived tables, it can be rebuilt from events.

### 4.6 Consolidator (the Sleep Cycle)

Largely unchanged from v0.0. One addition in v0.1:

**New sub-routine: Retrieval Trace Applier.** Runs on the light pass. Reads unprocessed `retrieval_trace` events, applies LTP updates to the referenced neurons and their co-retrieved synapses, inserts rows into the `retrievals` table, and marks events as processed (via a high-water-mark, not event mutation). This is the only place where retrieval-driven reinforcement happens in v0.1.

### 4.7 Retrieval (Associative Recall) — revised

**Purpose.** Given a query and a requestor scope, return the most relevant neurons + context, **without writing to any durable neuron state on the hot path.**

**Pipeline (v0.1).**

```
request(query, requestor_scope)
   │
   ▼
preprocess(query)
   │
   ├─► bm25.search()       ──┐
   ├─► vector.search()     ──┼─► rrf.fuse()  ─► top-K neurons
   └─► graph.walk()        ──┘
   │
   ▼
privacy.filter(scope)
   │
   ▼
redactor.finalize()
   │
   ▼
emit retrieval_trace event    ← cheap append, async from caller's perspective
   │
   ▼
response { neurons, synapses, episodes, citations, degraded_streams? }
```

**What changed.** In v0.0 the retrieval call performed a reconsolidation tick that updated `last_fired_at` and `fire_count` on top-K neurons synchronously. In v0.1, retrieval emits a single `retrieval_trace` event and returns. The consolidator picks up the trace on its next light pass and applies LTP updates in batch.

**Consequences.**

- Retrieval latency no longer includes a write transaction.
- Read replicas become feasible.
- LTP updates to the same popular neuron from concurrent retrievals are batched, not raced.
- Reinforcement is eventually-consistent with a lag bounded by the light-pass cadence (default 60s).
- The `retrieval_trace` event is itself a durable record usable by the healer (e.g. the `scope_leak_impossible` invariant).

**Event shape.**

```json
{
  "type": "retrieval_trace",
  "payload": {
    "request_id": "...",
    "query_hash": "...",
    "requestor_scope": "shared",
    "returned_neuron_ids": [123, 456, 789],
    "returned_max_scope": "shared",
    "degraded_streams": ["vector"]
  }
}
```

Note that the query itself is not stored — only its hash — so the trace does not become a new privacy surface.

### 4.8 Healing Layer

Unchanged from v0.0 in structure. Two invariants revised to reflect v0.1 semantics:

```yaml
- name: no_scrubber_drift_in_derived_state
  check: |
    SELECT count(*)
    FROM neurons n
    JOIN events e ON e.id = ANY(ARRAY(SELECT jsonb_array_elements_text(n.provenance->'event_ids')::bigint))
    WHERE e.scrubber_rev < :current_scrubber_rev
      AND n.superseded_by IS NULL
  expect: 0
  severity: medium
  repair: rederive_neurons_from_newer_scrubber

- name: scope_leak_impossible
  check: |
    SELECT count(*)
    FROM retrievals r
    WHERE r.ts > now() - interval '24 hours'
      AND scope_order(r.returned_max_scope) > scope_order(r.requestor_scope)
  expect: 0
  severity: critical
  repair: halt_and_alert
```

(v0.0 defined the `no_uncleaned_events` invariant to operate on `events`. That's incompatible with event immutability. v0.1 moves the invariant to the derived-state layer, which is the right place for it.)

The `rederive_neurons_from_newer_scrubber` repair:

1. Selects the affected neurons.
2. For each, fetches its provenance events, reruns the current scrubber on their payloads *in memory* (not writing back to events).
3. Produces new neuron rows (new IDs), copying provenance, with the freshly scrubbed content.
4. Supersedes the old neurons via `neuron_supersede` events.
5. Verifies the invariant now holds.

## 5. Data Flows

### 5.1 Ingest flow — revised

```
raw input
   │
   ▼
adapter.receive()
   │
   ▼
normalizer.canonicalize()
   │
   ▼
classifier.classify()
   │
   ▼
scrubber.scrub(scope)
   │
   ▼
compute content_hash
   │
   ▼
idempotency check
   ├─ key seen? → return existing event_id, STOP
   └─ new key (or no key) → continue
   │
   ▼
tombstone check
   ├─ content_hash in tombstones? → REJECT, STOP
   └─ clean → continue
   │
   ▼
events.append(event)
   │
   ▼
working_memory.ingest(event)
```

Note: no dedup-by-content. Two separate ingestions of the same content produce two events, and the consolidator treats that repetition as reinforcement.

### 5.2 Retrieval flow — revised

See §4.7. The retrieval call is pure-read; the LTP effect is applied asynchronously by the consolidator from the emitted `retrieval_trace`.

### 5.3 Consolidation flow

Unchanged from v0.0, plus the new Retrieval Trace Applier sub-step on each light pass.

### 5.5 Unlearn / Relearn (Reconsolidation Protocol) — revised

v0.0 described this as "in-place update," which misrepresented how the state change is durable. v0.1 clarifies: **every neuron mutation is accompanied by an event.** The neuron row is updated (with optimistic concurrency via `version`), and in the same transaction an event is appended to the log describing the mutation. Replay of the event log reproduces the neuron's full history.

Three mutation events exist:

- `neuron_reinforce` — bumps activation, fire_count, confidence (LTP).
- `neuron_update` — content/provenance modified during reconsolidation window.
- `neuron_supersede` — creates a successor neuron and points the predecessor at it.

Protocol:

```
on retrieval of neuron N (via retrieval_trace → consolidator):
    emit neuron_reinforce for N
    N.state becomes labile for RECONSOLIDATION_WINDOW (unjustified default: 5 minutes)

while N is labile:
    on new event E concerning N's subject:
        if E supports N:
            emit neuron_reinforce for N (stronger: α > normal retrieval bump)
        elif E contradicts N:
            create N' with E's content
            emit neuron_supersede (old=N, new=N')
            N is not deleted; still readable with include_superseded=true
        elif E refines N:
            merged_content = llm.merge(N.content, E.content)
            emit neuron_update(N, merged_content, provenance += E.id)

when window closes:
    N becomes stable
    if N was not reinforced during window:
        decayer applies mild LTD on next decay tick (not on close)
```

**Concurrency model (new in v0.1).** Neuron updates use optimistic concurrency. Every write is:

```
SELECT version FROM neurons WHERE id = N.id
-- compute update U
UPDATE neurons SET ..., version = version + 1
  WHERE id = N.id AND version = :expected_version
-- if 0 rows affected → conflict → re-read, re-apply, retry (bounded)
```

Conflicting updates within the reconsolidation window resolve by serialization through the version token; each successful update emits its own event, preserving full history in the log. There is no silent last-write-wins.

**Hard-delete (protective forgetting)** remains unchanged from v0.0: tombstone the `content_hash`, delete the neuron and its synapses, delete the originating events, log the action (not the content) to `healing_log`, re-run affected invariants.

## 6. Privacy Model

### 6.1 Scope Lattice

`public  ≤  shared  ≤  private`. "Higher" means more restrictive.

Every neuron has exactly one scope. A retrieval requestor presents its own scope. The enforcer returns only neurons where `neuron.scope ≤ requestor.scope`.

### 6.2 Scope Assignment — rewritten in v0.1

v0.0 contained a self-interrupting sentence. The intended rules are:

- **Automatic tightening is always allowed.** If the classifier or scrubber finds signals that content is more sensitive than declared (PII detected, secrets found, source is flagged private), the neuron's scope is raised toward `private` automatically.
- **Automatic loosening is never allowed.** No piece of ingest pipeline may move scope toward `public` without explicit human/operator action.
- **Loosening requires an explicit re-classification event.** A caller with authority emits a `reclassify_scope` event citing the neuron, the new scope, and a reason. The healer verifies the caller's authority before applying.
- **Ceiling is `private`.** Nothing goes higher.
- **Defaults are conservative.** Unclassified content defaults to `private`. It may only become `shared` or `public` via explicit classification with evidence.

## 12. Testing Strategy

Unchanged from v0.0, plus:

- **Reinforcement test.** Ingest the same content N times. Assert N events appended, one neuron exists (after consolidation), `source_count ≥ N` and `activation` increased accordingly.
- **Idempotency test.** POST the same ingest with the same `idempotency_key` N times. Assert one event appended, N identical responses returned.
- **Retrieval-is-read-only test.** Run `/v1/recall` under a read-only DB connection. Assert it succeeds (retrieval_trace is emitted via a separate event-log writer, not via the retrieval DB session). This is a structural test that prevents regressions on the v0.1 hot-path rule.
- **Optimistic-concurrency test.** Concurrently apply two `neuron_update`s to the same labile neuron. Assert both succeed (one on first try, one on retry), both events land, final state is deterministic given event order.

## 16. Governance — revised

The eight contracts from v0.0 still stand. Two are restated more strictly in v0.1:

1. **Events are immutable.** Once appended, an event's row is never modified. Protective forgetting deletes; nothing edits. *(v0.1 strengthens: no carve-out for rescrubbing.)*
2. **Derived state is disposable.** Neurons, synapses, episodes, patterns, retrievals can always be rebuilt from events.
3. **Scope can be tightened automatically; loosening requires an explicit re-classification event.** *(v0.1 restates the v0.0 rule without the self-interrupting sentence.)*
4. **Secrets never appear in embeddings.**
5. **Invariants are declarative.**
6. **Provenance on everything.**
7. **Retrieval never writes to neuron state synchronously.** *(v0.1 new — was violated by v0.0.)* Retrieval may append a `retrieval_trace` event to the log. Durable neuron mutations happen only in the consolidator.
8. **Every neuron mutation emits an event.** *(v0.1 new — makes the event-sourcing guarantee complete.)* The neuron row is updated under optimistic concurrency, and the mutation event is appended atomically. Replay reproduces neuron state.
9. **Single writer per table.**

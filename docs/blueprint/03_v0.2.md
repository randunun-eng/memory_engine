## 3. Biology → Engineering Mapping — rewritten framing

The table of biological analogies (neuron, synapse, LTP, LTD, Hebbian learning, hippocampal consolidation, reconsolidation, BBB) still appears in v0.0. What changes in v0.2 is how we talk about it.

**v0.1 said:** "This table is the contract. If a proposed change breaks one of these mappings, the design is wrong."

**v0.2 says:** This table is the design vocabulary. It is a source of intuition and a consistent naming convention — nothing more. Engineering decisions stand on engineering grounds (correctness, latency, cost, operational simplicity), not on whether they preserve an analogy.

Concretely:

- If an engineering constraint forces a design that breaks the Hebbian analogy, the engineering wins. We update the vocabulary.
- The analogies are not predictive. Real neurons do not have `version` columns; we have them because optimistic concurrency requires them. Do not reach for biology to justify an engineering choice.
- Numbers in biology (5-minute reconsolidation windows, forgetting-curve half-lives) are inspiration for defaults, not grounds for them. Defaults get tuned by measurement — see §10 and §18.
- Where biology and engineering disagree on vocabulary, engineering wins. "Synaptic pruning" is a fine colloquial name for "GC of low-weight edges." If someone reads the code and is confused, we rename.

The reason for this walk-back: in v0.0 the biology language was being used as a governance tool ("breaks the mapping → design is wrong"), which is a category error. Metaphors are not theorems. Governance lives in §16.

## 4.6 Consolidator — procedural-tier spec (revised)

v0.1 said the pattern inducer detected "sub-graphs appearing in M+ episodes" and promoted them. That sentence was handwaving: general subgraph matching is NP-hard, and the spec gave no concrete schema for what a pattern even looks like.

v0.2 replaces pattern induction with **skill crystallization**, borrowed from GenericAgent and adapted to a memory-layer context.

### What a skill is in a memory layer

A memory layer's procedural memory is not agent workflows (we're not an agent). It is **retrieval strategies**: recurring query shapes where a specific retrieval approach beats the generic hybrid-search default.

Examples of retrieval strategies that warrant crystallization:
- "Queries of shape 'what did person X say about topic Y'" → specific graph walk starting at person node, filtered by `speaks_about` edges, bounded to last 20 episodes.
- "Queries of shape 'project Z status'" → episode-first retrieval sorted by recency, then expand to related neurons.
- "Queries matching a regex pattern for date ranges" → event-log scan within range, skip semantic layer.

These are strategies the retrieval layer can learn from what actually works.

### Schema

```sql
skills(
  id                  BIGSERIAL PRIMARY KEY,
  name                TEXT NOT NULL,
  trigger_pattern     JSONB NOT NULL,     -- conditions: query shape, entity types, keywords
  retrieval_plan      JSONB NOT NULL,     -- concrete steps: which streams, which filters, which walks
  success_metric      JSONB NOT NULL,     -- how we measure whether the skill performed well
  support_count       INT NOT NULL,       -- how many successful applications
  failure_count       INT NOT NULL,
  confidence          REAL NOT NULL,      -- support / (support + failure) with smoothing
  provenance_episodes BIGINT[] NOT NULL,  -- which episodes the skill was derived from
  created_at          TIMESTAMPTZ NOT NULL,
  last_fired_at       TIMESTAMPTZ NOT NULL
)
```

### How skills are created

1. The observer logs every retrieval with its query, chosen strategy, and outcome signals (was the result reinforced later? did the user issue a follow-up that suggests the recall was wrong?).
2. The consolidator, on its deep pass, clusters recent retrievals by query shape (features: entity types mentioned, temporal markers, keywords, requestor scope).
3. For each cluster that has N+ retrievals with a consistent successful strategy, a candidate skill is proposed.
4. The skill goes through a shadow period (default 50 retrievals) where it runs alongside the default strategy and its results are compared. Only if it beats default does it activate.
5. Active skills are applied to matching queries; their success signals update `support_count` / `failure_count`.

### How skills are retired

- Confidence below `skill_retire_threshold` (default 0.4) for N+ consecutive evaluations → retire.
- `last_fired_at` older than retention window → archive (can be revived if matching queries return).

### Why this is better than v0.1's pattern inducer

- Well-defined inputs (retrieval logs) and outputs (a typed row).
- No NP-hard graph matching. Clustering on features is standard.
- Every skill has a measurable success signal, so skills that hurt performance die automatically.
- Shadow-period gating means skills can't silently make retrieval worse.
- Borrowed from GenericAgent with the one change that matters: we're crystallizing *retrieval strategies*, not *agent tool sequences*.

### What we did not borrow from GenericAgent

- Tool-level skills (code_run, file_write, etc.) — we're not an agent.
- Bot-interface abstractions (QQ, Telegram, Feishu adapters) — out of scope.
- The "~3,300 lines of code" framing — code size is not a design goal.
- Memory tiers L0/L2/L3 — we already have four tiers defined more carefully.

## 11. Implementation Roadmap — honest estimates

v0.0 said 8 weeks at half-time for a solo developer. That was optimistic. Revised:

| Phase | v0.0 estimate | v0.2 estimate (half-time solo) | What slips first |
|---|---|---|---|
| 0 — Skeleton | 1 wk | 2 wk | Migrations runner, schema decisions |
| 1 — Ingest + Working | 1 wk | 2 wk | PII scrubber rule coverage |
| 2 — Semantic + Retrieval | 1–2 wk | 3–4 wk | Hybrid fusion tuning, LLM prompt stability |
| 3 — Decay / Contradictions | 1 wk | 2 wk | Contradiction judge reliability |
| 4 — Healing | 1 wk | 2 wk | Invariant dry-run → enable flow |
| 5 — Procedural (skills) | 1 wk | 2 wk | Skill shadow-period plumbing |
| 6 — Observability | 1 wk | 1–2 wk | — |
| 7 — Eval harness | *(was not in v0.0)* | 2 wk | Benchmark adapters |
| 8 — Docs + release | 1 wk | 1–2 wk | — |

**Honest total: 17–22 weeks at half-time (340–440 hours).** Full-time solo: 8–10 weeks if the developer has prior event-sourcing and memory-system experience, 12–16 weeks otherwise.

**What slips first, in order of probability.**

1. **Phase 2 (retrieval)** — RRF tuning is iterative. Plan for 2 extra weeks, not zero.
2. **Phase 5 (skills)** — the shadow-period comparison harness is more work than it sounds.
3. **Phase 7 (eval harness)** — LongMemEval adapters require writing glue; MemoryBench integration likewise.
4. **Any phase that touches LLM prompts** — prompt reliability eats time in unpredictable chunks.

**What is cheaper than it looks.**

- Phase 0 and 6 (pure plumbing) are cheap if you've done similar systems before.
- Phase 4 (healing) is cheap if you commit to dry-run-before-enable for every new invariant. Skip that discipline and it becomes expensive.

**Rule of thumb:** if a phase involves tuning a threshold or a prompt, double the estimate.

## 18. Evaluation Framework (new in v0.2)

Without evaluation, every memory system drifts. v0.2 adds a concrete framework.

### 18.1 What we measure

Three categories of metrics, each with targets and degraded-mode alerts.

**Retrieval quality.**

| Metric | Definition | Target (v0.1 ship bar) |
|---|---|---|
| Recall@10 | Of the known-relevant neurons for a query, how many in top 10? | ≥ 0.70 |
| MRR (Mean Reciprocal Rank) | Average of 1/rank of first relevant result | ≥ 0.50 |
| NDCG@10 | Rank-aware quality | ≥ 0.60 |
| p50 latency | Median retrieval time | ≤ 150 ms |
| p99 latency | Tail latency | ≤ 800 ms |

**Memory lifecycle.**

| Metric | Definition | Target |
|---|---|---|
| Supersession correctness | Of manually-labeled contradiction pairs, how often does the healer supersede correctly? | ≥ 0.85 |
| Decay calibration | After retention window, does activation match expected curve? | within ±15% of expected |
| Dedupe rate | Fraction of duplicate content correctly reinforced (not re-inserted as new neurons) | ≥ 0.95 |
| Skill shadow accuracy | Fraction of shadow-period skill comparisons where the winning strategy is actually better (measured by held-out recall) | ≥ 0.80 |

**Safety and reliability.**

| Metric | Definition | Target |
|---|---|---|
| Scope leak rate | Private neurons returned to non-private requestor in 10k retrievals | **0** (hard) |
| PII leak rate | PII patterns found in egress payloads over 10k retrievals | **0** (hard) |
| Invariant MTTR | Mean time from invariant violation to repair | ≤ 5 minutes |
| Unclean shutdown recovery | Ingest rate 60s after `kill -9` of the process | ≥ 95% of steady-state |
| Replay determinism | Derived state after full replay of events equals original state | bit-exact for structured fields |

### 18.2 Eval datasets

**Internal (synthetic, shipped with repo).**

A generator produces event streams with known structure: known duplicates, known contradictions, known entity relationships. The generator seeds with a random seed so runs are reproducible. No real data in repo.

- ~10k events per run.
- 500 retrieval test queries with labeled expected results.
- 100 contradiction pairs with labeled supersession direction.
- 50 privacy test cases (private neuron, multiple requestor scopes).

**External (adapters, not shipped).**

- **LongMemEval** — published memory benchmark, ~500 questions requiring multi-session recall. Adapter translates LongMemEval sessions to our event log and queries to our `/v1/recall`. Useful for comparison with other systems.
- **MemoryBench** — similar. Newer benchmark. Adapter optional.

External benchmarks are adapters, not requirements. Internal eval must pass for every PR to `main`.

### 18.3 Eval harness

```
wiki-v3 eval synth            # runs internal synthetic eval, prints report
wiki-v3 eval longmemeval      # runs LongMemEval if data present
wiki-v3 eval compare v0 v1    # compares two runs; flags regressions
```

CI runs `wiki-v3 eval synth` on every PR. If any hard-zero metric (scope leak, PII leak) is non-zero, the PR is blocked. If any target metric regresses by more than 5% versus `main`, the PR requires manual justification.

### 18.4 What eval does not cover

- Real-world subjective quality ("does it feel like it remembers me"). Synthetic eval cannot judge this. Ship with honest disclaimer.
- Long-tail failure modes discovered in production. These get added to the eval set after the fact, not before.
- Multi-agent interaction. Non-goal for v0.1 (see §13).

## 19. Cost Model (new in v0.2)

LLM costs in v0.1 were unbounded and unaddressed. v0.2 makes them first-class.

### 19.1 Where LLMs are called

| Call site | Frequency | Cost sensitivity |
|---|---|---|
| Scope classification (ingest) | Per event | High — runs on every input |
| Entity extraction (promotion) | Per session close | Medium |
| Episode summarization | Per session close | Medium |
| Contradiction judging | Per near-duplicate during deep pass | Medium |
| Content merging (reconsolidation) | Per labile-window update | Low |
| Skill shadow evaluation | Per shadow retrieval | Low |

Naive implementation of all six, on 10k events/day with a commercial LLM, lands between $50 and $500/month depending on model choice. That's not acceptable as a default.

### 19.2 Cost controls

**Tier 1 — Skip LLM when possible.**

- Scope classification: if regex-based rules produce confidence > `scope_rule_confidence` (default 0.9), skip LLM call.
- Entity extraction: run NER first (spaCy or similar local); only call LLM for entity types NER doesn't cover or when the episode is flagged complex.
- Summarization: skip for sessions shorter than `min_tokens_for_llm_summary` (default 500); template-based summary suffices.

**Tier 2 — Use local models first.**

Default config uses a local model (Qwen3 / Llama 3 class) for classification and routine extraction. Remote model is used only for:
- Contradiction judging (requires nuance).
- Content merging.
- Explicit "deep" requests via API flag.

Config knobs:
```toml
[llm.routing]
classification_model = "local"       # local | remote
extraction_model = "local"
summarization_model = "local"
judging_model = "remote"             # decision: nuance matters here
merging_model = "remote"

[llm.remote]
provider = "configurable"            # no vendor baked in
model = "configurable"
```

**Tier 3 — Cache.**

- Response cache keyed on `(call_site, prompt_hash, model_id, prompt_template_rev)`.
- Default TTL: 30 days.
- Cache invalidation on `prompt_template_rev` bump.
- Cache is local SQLite; no remote round-trip for repeated prompts.

**Tier 4 — Batch.**

Consolidator deep pass batches related work into fewer calls: e.g. extract entities for 10 episodes in one prompt rather than 10.

**Tier 5 — Budget enforcement.**

```toml
[costs]
monthly_budget_usd = 0               # 0 = local only
budget_enforcement = "degrade"
```

Modes:
- `ignore` — no enforcement; log spend only.
- `degrade` — on budget exhaustion, fall back to local-only models; mark affected neurons as `degraded_provenance` so healing can reconsider them when budget resets.
- `halt` — reject new ingest once budget exhausted; alert.

Default is `monthly_budget_usd = 0`, which forces local models and costs nothing. Operators opt in to remote spend consciously.

### 19.3 Cost observability

`/metrics` exposes:
- `wiki_v3_llm_cost_usd_total{site, model}` — running spend.
- `wiki_v3_llm_calls_total{site, model, cache_hit}` — call counts.
- `wiki_v3_llm_tokens_total{site, model, direction}` — token counts.

`wiki-v3 doctor --costs` prints a 30-day breakdown.

### 19.4 Cost budget for v0.1 ship

With defaults (local model, cache enabled, Tier 1 skip rules on):

- Ingest: **$0** (all local).
- Deep pass: **$0** (all local).
- Optional remote judging/merging: typically < $0.01 per 1000 contradiction pairs judged with a mid-tier model.

The system is usable at zero cost on a workstation. Remote spend is a choice for users with specific quality requirements, not a default.

## 20. Log Compaction & Archival (new in v0.2)

The event log grows unboundedly. At 10k events/day, a year is 3.6M events. Replay from epoch becomes unusable. v0.1 punted on this; v0.2 specifies a strategy.

### 20.1 Governance first

Events are never truncated by default. "Replay from events" remains a guarantee. This is an explicit governance rule (added to §16 as rule 10).

But replay can start from a snapshot, not from epoch.

### 20.2 Snapshots

A **snapshot** is a consistent copy of all derived tables (neurons, synapses, episodes, skills, retrievals) plus the `events.id` at which the snapshot was taken.

```sql
snapshots(
  id               BIGSERIAL PRIMARY KEY,
  taken_at         TIMESTAMPTZ NOT NULL,
  up_to_event_id   BIGINT NOT NULL,      -- snapshot reflects events with id <= this
  derived_digest   BYTEA NOT NULL,       -- hash of all derived tables at snapshot time
  compression      TEXT NOT NULL,        -- 'zstd' | 'none'
  path             TEXT NOT NULL         -- on-disk location of the snapshot blob
)
```

**Snapshot cadence.**
- Automatic: every N events (default 100_000, guess) or every T hours (default 24, guess), whichever first.
- Manual: `wiki-v3 snapshot create`.
- Retention: keep last K snapshots (default 7). Older snapshots deleted automatically unless pinned.

**Replay semantics.**

`wiki-v3 replay --to <event_id>` now works as:
1. Find latest snapshot with `up_to_event_id <= target`.
2. Restore derived tables from snapshot.
3. Apply events from `snapshot.up_to_event_id + 1` to `target`.
4. Verify `derived_digest` if replay target matches a snapshot.

Replay time goes from O(all events) to O(events since last snapshot).

### 20.3 Archival

Optional. For deployments that need to keep very old events but want them out of the hot database.

**Archive tier.** Events older than `archive_after_days` (default: disabled) are moved to cold storage (compressed parquet files on disk or object store). The row in `events` is replaced with a placeholder that records the archive location.

Archived events are still replayable but slower. Snapshots cover the common case; archive is for audit and forensics.

```toml
[archive]
enabled = false                      # decision: off by default
after_days = 365                     # guess
backend = "local"                    # local | s3 | configurable
path = "./data/archive"
```

### 20.4 What this does not include

- Automatic archive-to-cloud. User configures.
- Differential snapshots. Full only in v0.1. Differential can come later.
- Point-in-time recovery. Snapshots + replay give this implicitly.

## 16. Governance — additions in v0.2

v0.0 had 8 governance rules. v0.1 added rules 7 and 8. v0.2 adds rule 10. (Rule 9 was "single writer per table," already in v0.1.)

**Rule 10.** Events are never truncated by default. Replay from the full event log must always be possible, even if slow. Snapshotting (§20) is an optimization, not a replacement.

Rationale: the one guarantee the system offers operators is "any derived state can be rebuilt from events." Allowing truncation would break that guarantee silently. Operators who explicitly enable archival accept the trade-off; the default does not.

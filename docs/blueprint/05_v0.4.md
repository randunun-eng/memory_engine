# Wiki v3 — The Neural Memory Engine (v0.4)

> Companion document to v0.0 + v0.1 + v0.2 + v0.3. Read those first.
> v0.4 is the anti-hallucination revision. It closes gaps exposed by Graphiti's bi-temporal modeling and by a public production audit of mem0 that found 97.8% junk in a 32-day deployment.

---

## Revision notes — what changed from v0.3

v0.3 built the identity layer. v0.4 hardens the substrate against hallucination. Prompted by two external findings:

1. **Graphiti (Zep)** uses bi-temporal modeling — separating "when was this true in reality" from "when did the system learn it." v0.3 only tracked the latter. This is a real gap for digital twins.
2. **mem0 production audit** (public issue #4573) found one extraction hallucination reinforced into 808 copies over 32 days because the pipeline had no grounding verification before storage. Our v0.3 consolidator would exhibit the same failure mode: the reinforcement-on-repeat-content-hash mechanism amplifies any extraction error.

v0.4 addresses both. Seven targeted changes:

| # | Change | Addresses | Section |
|---|---|---|---|
| 1 | Bi-temporal modeling: add `t_valid`, `t_invalid` alongside existing `superseded_at` | Point-in-time queries; "Alice worked at Acme from March to September" vs "I learned about it on Tuesday" | §4.5 revised |
| 2 | Grounding verification gate before event → neuron promotion | Prevents extraction hallucinations from entering memory (mem0 lesson) | §4.1 revised, §4.6 revised |
| 3 | `distinct_source_count` alongside `source_count` | One hallucinated extraction reinforced 807 times should not look like 808 sources | §4.5 revised |
| 4 | Targeted contradiction detection (constrained to same entity-pair) | Cheaper, fewer false positives (Graphiti pattern) | §4.6 revised |
| 5 | Identity-aware extraction (extractor told whether subject is human/agent/group/system) | mem0 hallucinated physical demographics for a software process because the extractor didn't know | §4.6 revised |
| 6 | Point-in-time query support: `as_of` parameter on recall | Enables "what did I think about X last month" queries without rebuilding | §4.7 revised |
| 7 | Eval targets recalibrated to honest SOTA | HaluMem shows all systems <62% extraction accuracy, <70% QA; v0.2's 85% targets were fiction | §18 revised |

The reinforcement model changes. v0.3 said repeated `content_hash` is a valid reinforcement signal. v0.4 qualifies that: repetition strengthens a neuron only when the repetitions come from distinct source events, not from a consolidator loop echoing one extraction. This is the biggest behavioral change in v0.4.

## 4.5 Storage — bi-temporal schema (revised)

### Neurons: add validity-time dimension

```sql
neurons(
  id                    BIGSERIAL PRIMARY KEY,
  persona_id            BIGINT NOT NULL REFERENCES personas(id),
  kind                  TEXT NOT NULL,
  form                  TEXT NOT NULL,
  counterparty_id       BIGINT REFERENCES counterparties(id),
  tier                  TEXT NOT NULL,
  content               TEXT NOT NULL,
  embedding             VECTOR(N),
  embedder_rev          INT NOT NULL,
  activation            REAL NOT NULL DEFAULT 0.5,
  confidence            REAL NOT NULL DEFAULT 0.5,
  scope                 TEXT NOT NULL,

  -- transactional timeline (when the system learned/invalidated)
  created_at            TIMESTAMPTZ NOT NULL,
  last_fired_at         TIMESTAMPTZ NOT NULL,
  superseded_by         BIGINT REFERENCES neurons(id),
  superseded_at         TIMESTAMPTZ,

  -- validity timeline (when this was true in reality) — NEW in v0.4
  t_valid_start         TIMESTAMPTZ,       -- NULL = always / unknown
  t_valid_end           TIMESTAMPTZ,       -- NULL = still true / unknown

  -- reinforcement accounting (revised in v0.4)
  fire_count            INT NOT NULL DEFAULT 0,
  source_count          INT NOT NULL DEFAULT 1,   -- total events supporting (may include echoes)
  distinct_source_count INT NOT NULL DEFAULT 1,   -- events from genuinely distinct sources (NEW)

  provenance            JSONB NOT NULL,
  version               INT NOT NULL DEFAULT 0,
  CHECK (...)                                     -- as v0.3
);

CREATE INDEX ON neurons (persona_id, t_valid_start, t_valid_end);  -- for as-of queries
```

### Synapses: same treatment

Edges (synapses) also gain `t_valid_start` / `t_valid_end`. A "works_at" synapse between Alice and Acme may have `t_valid_start = 2025-03-01, t_valid_end = 2025-09-15`, preserved as a historical fact even after Alice changes jobs.

### What validity-time means and does not mean

- **Populated when extractable.** If the source text says "Alice started at Acme in March 2025," extraction sets `t_valid_start = 2025-03-01`. If it says "Alice works at Acme" with no date, `t_valid_start` is NULL (unknown, treat as "since first observation").
- **Never fabricated.** If the extractor cannot cite a specific temporal claim in the source, these columns stay NULL. The extractor prompt (§4.6) is explicit about this.
- **Queries respect both timelines.** A "what do I know about Alice" recall defaults to "currently true": `WHERE (t_valid_end IS NULL OR t_valid_end > now()) AND superseded_by IS NULL`. An "as of T" recall uses `t_valid_start <= T AND (t_valid_end IS NULL OR t_valid_end > T)`.

### Distinct-source tracking — why it matters

Under v0.3, the consolidator on every deep pass saw recurring content hashes and bumped `source_count`. If an extraction hallucinated "User prefers Vim" and the same downstream pipeline re-extracted it from the same originating event on every subsequent pass, `source_count` climbed. The neuron looked strongly supported. It wasn't.

Under v0.4, every reinforcement emits the `source_event_ids` it is claiming as distinct sources. The reinforcer checks: is this `source_event_id` already in the neuron's provenance? If yes, `source_count` still increments (for audit) but `distinct_source_count` does not. Retrieval ranking uses `distinct_source_count`, not `source_count`, for confidence weighting.

This is one line of code and one integer column. It is the single most important anti-hallucination change in v0.4.

## 4.1 Sensory Layer — grounding verification gate (revised)

The mem0 failure mode had no filter between extraction and storage. v0.4 adds one.

### Gate placement

```
existing ingest pipeline (v0.1/v0.3):
  raw → normalize → classify → scrub → dedup check → append event
                                                        │
                                                        ▼
                                              (consolidator later extracts neurons)

v0.4 ingest pipeline (unchanged for raw events):
  raw → normalize → classify → scrub → dedup check → append event
                                                        │
                                                        ▼
                                              consolidator extracts candidate neurons
                                                        │
                                                        ▼
                                              ┌─ GROUNDING GATE (new in v0.4) ─┐
                                              │ verify each candidate cites    │
                                              │ specific source event IDs;     │
                                              │ re-check that source supports  │
                                              │ the extracted claim            │
                                              └────────────────┬───────────────┘
                                                               │
                                              ┌────────────────┴───────────────┐
                                              │ pass → persist neuron          │
                                              │ fail → quarantine + alert      │
                                              └────────────────────────────────┘
```

The gate sits between extraction and storage. Raw events are still appended unconditionally — they are ground truth. The gate only governs what becomes a neuron.

### Gate rules

A candidate neuron is accepted if all of:

1. **Citation present.** Candidate must name at least one `source_event_id` from the batch that produced it.
2. **Citation resolves.** Named event(s) exist and belong to the current persona.
3. **Citation supports claim.** A cheap verification: does the candidate's content appear semantically grounded in the cited event's payload? Default: embedding similarity between claim and cited event content above threshold (`grounding_similarity_min`, default 0.5 — guess, tune). For high-confidence tiers, a second LLM-judge pass. Failures here are not automatic rejections; they go to the review queue with the candidate, the claimed citation, and the similarity score.
4. **Claim is specific, not generic.** Candidates that are essentially restatements of the persona's role or the counterparty's identity ("user is a person who sends messages") are rejected as non-informative. Pattern library maintained in `config/grounding_generic_patterns.yaml`.

### Quarantine, not silent drop

Rejected candidates do not disappear. They go to a `quarantine_neurons` table with:
- The proposed neuron content
- The citations claimed
- The reason for rejection
- The grounding score

Operator reviews the quarantine queue periodically. Frequent patterns (e.g., a specific extraction prompt producing consistent rejections) indicate the extractor needs prompt revision. Low-volume quarantine = healthy system; growing quarantine = upstream quality problem.

### Cost

One embedding comparison per candidate: cheap. One LLM-judge pass for high-confidence tiers: moderate; cached by (claim_hash, cited_events_hash). Overall: gate adds ~5-15% to consolidator cost under defaults, much less with caching warm. Documented in §19 cost update.

## 4.6 Consolidator — identity-aware extraction and targeted contradiction (revised)

### Identity-aware extraction prompts

v0.3 extraction prompts treated every subject as generic. Result (mem0-style): extractor fabricates demographics because the prompt doesn't know the subject is an AI agent, a group, or a company.

v0.4 extraction prompts receive structured subject metadata:

```
You are extracting facts from a conversation.
Subject: whatsapp:+94771234567
Subject kind: HUMAN_INDIVIDUAL
Persona: randunu-twin
Persona kind: HUMAN_PERSONAL_TWIN
Counterparty context: ACQUAINTANCE, first seen 14 days ago

Rules:
- Extract only claims the source text directly supports.
- If the subject kind is not HUMAN_INDIVIDUAL, do not extract demographic attributes (age, location, gender, occupation) unless explicitly stated.
- If the subject kind is AI_AGENT or SYSTEM, do not extract preferences, emotions, or intentions.
- For every extracted fact, cite the exact message or span that supports it.
- If a claim cannot be grounded in a specific span, do not emit it.
```

The `_kind` enumerations are configurable per deployment. Default set:
- `HUMAN_INDIVIDUAL`, `HUMAN_GROUP`, `HUMAN_ORGANIZATION`, `AI_AGENT`, `SOFTWARE_SYSTEM`, `UNKNOWN`

Counterparties get kind-tagged on first creation via a lightweight classifier (local model, regex on external_ref, user metadata). The tag can be updated by operator; never silently.

### Targeted contradiction detection

v0.3 called `llm.contradicts(new_neuron, similar_neurons)` where "similar" came from general vector search. Expensive and noisy.

v0.4 constrains: for a new synapse `(entity_A, rel, entity_B)`, look for existing synapses with the same `(pre_id, post_id)` pair. Only those are candidates for contradiction check. For a new neuron attribute on entity E, look for neurons about entity E with overlapping `t_valid` intervals.

This is the Graphiti pattern. Two consequences:
- The LLM judge runs on a tiny candidate set (usually 0-5 existing edges) rather than dozens.
- False positives drop because unrelated-but-semantically-similar facts no longer trigger comparison.

When contradiction is detected, the resolution path uses `t_valid` — the newer-true fact wins for "currently true" queries but the older fact keeps its validity interval intact for "as of" queries. This is the non-lossy invalidation Graphiti pioneered.

### Consolidator deep-pass pseudocode (v0.4)

```
for each candidate_neuron in batch:
    # Grounding gate (§4.1)
    if not grounding_gate.verify(candidate_neuron):
        quarantine(candidate_neuron, reason)
        continue

    # Dedup against existing
    similar = vector_search(candidate_neuron.embedding, same_persona, same_entity)
    match = find_exact_or_near(similar, candidate_neuron)

    if match and candidate_neuron.source_event_id in match.provenance.event_ids:
        # Echo of already-processed event — do not reinforce distinct
        match.source_count += 1
        # match.distinct_source_count NOT incremented
    elif match:
        # Genuine new support from new source
        match.source_count += 1
        match.distinct_source_count += 1
        match.provenance.event_ids.append(candidate_neuron.source_event_id)
        if candidate_neuron.t_valid_start and not match.t_valid_start:
            match.t_valid_start = candidate_neuron.t_valid_start
    else:
        # New neuron
        persist(candidate_neuron)

# Contradiction detection (targeted)
for each new_synapse:
    competitors = find_same_entity_pair_edges(new_synapse.pre_id, new_synapse.post_id)
    competitors = filter_by_overlapping_validity(competitors, new_synapse.t_valid_start)
    for c in competitors:
        verdict = llm.judge_contradiction(new_synapse, c)
        if verdict == "contradicts":
            c.t_valid_end = new_synapse.t_valid_start or now()
            # c is NOT deleted; it's historically valid
```

## 4.7 Retrieval — as-of queries (revised)

New optional parameter on `/v1/recall`:

```json
{
  "query": "what do I know about alice",
  "lens": "counterparty:alice",
  "requestor_scope": "private",
  "as_of": "2025-09-01T00:00:00Z"    // NEW: point-in-time query
}
```

Semantics:
- `as_of` omitted → current state (default). Returns neurons where `(t_valid_end IS NULL OR t_valid_end > now()) AND superseded_by IS NULL`.
- `as_of` present → returns state as of that moment in reality. Uses validity-time: `t_valid_start <= as_of AND (t_valid_end IS NULL OR t_valid_end > as_of)`.

Requires `t_valid_start` / `t_valid_end` indexing. For deployments that never set validity times, `as_of` degrades to "neurons created before `as_of` that were not yet superseded then" — less accurate but functional.

This enables use cases like:
- "What did I think about this customer's account last month?"
- "What was Alice's job at the time of that meeting?"
- "Replay my understanding as of the day of the incident."

Without bi-temporal modeling these queries are either impossible or lie.

## 18. Evaluation — honest SOTA calibration (revised)

v0.2 §18 targeted 0.70 Recall@10, 0.50 MRR, 0.60 NDCG@10. Those numbers were written without reference to actual memory-system benchmarks.

HaluMem, a 2025 benchmark targeting hallucination in memory systems, shows the state of the field: all evaluated systems (Mem0, Mem0-Graph, MemOS, Memobase, Supermemory) score **below 62% accuracy on memory extraction** and **below 70% on QA**, with substantial declines on longer contexts. That's current SOTA, not a strawman.

### Recalibrated targets

| Metric | v0.2 target | v0.4 target | Rationale |
|---|---|---|---|
| Extraction accuracy (grounding-verified) | implicit 100% | ≥ 75% | Grounding gate should beat SOTA's 62% because we reject ungrounded extractions; 75% is ambitious but defensible |
| QA answer accuracy on internal eval | not measured | ≥ 60% | Below the best reported system; we claim grounding discipline, not algorithmic superiority |
| Hallucinated-fact rate in retrieved context | implicit 0% | ≤ 5% | Hard targets of 0% are aspirational; 5% is a credible operating point |
| Reinforcement echo rate | not measured | ≤ 1% | Fraction of neurons where `source_count > 1` but `distinct_source_count = 1` (mem0-style echoes); direct measure of our biggest fix |
| Point-in-time query correctness | not measured | ≥ 90% | Queries with `as_of` return state that matches validity intervals |
| Recall@10, MRR, NDCG@10 | as v0.2 | as v0.2 | Retrieval quality metrics unchanged |
| Scope/PII/cross-counterparty leak rates | 0 hard | 0 hard | Unchanged |

The hard zeros (privacy, cross-counterparty) remain hard. The accuracy targets become honest.

### New eval: extraction grounding

The synthetic fixture from v0.2 gains a category:

- 100 "ungrounded extraction temptation" cases: conversations where a weak extractor would fabricate (names mentioned in passing, demographics implied but not stated, counterfactual phrases like "imagine if…"). Target: grounding gate rejects ≥ 90% of these in the consolidator.

And a regression test derived directly from the mem0 audit:

- Ingest 50 identical echo events of one extracted fact. Verify: `source_count` climbs to 50, `distinct_source_count` stays at 1, retrieval ranking treats the neuron as 1-source evidence.

The second test is the ~20-line assertion that would have prevented the 808-Vim-copies problem.

## 19. Cost model — v0.4 additions

Grounding gate adds LLM calls:

- **Embedding similarity check (default path).** Reuses the candidate's embedding and cited event's embedding (both already computed). Near-zero marginal cost.
- **LLM-judge pass (for high-confidence tier only).** One call per candidate that passes embedding similarity but needs deeper verification. Cached by `(claim_hash, cited_events_hash)` with long TTL.

At default config (local classifier, cached judge calls warm), grounding gate adds **<5% to per-event LLM cost**. With a remote judge model and cold cache, **up to 20%**. Both still fit within v0.2 §19's zero-cost default.

Identity-aware extraction adds no calls; it's a prompt change.

Targeted contradiction detection **reduces** LLM calls — the candidate set shrinks from "general similar neurons" (dozens) to "same entity-pair edges" (usually 0-5). Net cost impact: negative.

## 16. Governance — additions in v0.4

Rules 1–13 from v0.0/v0.1/v0.2/v0.3 stand. v0.4 adds:

**Rule 14.** Every neuron has at least one citation to a specific source event. Neurons whose citations cannot be resolved are invalid and must be quarantined.

**Rule 15.** Reinforcement by `distinct_source_count` uses genuinely distinct source events. Reinforcement by `source_count` is for audit only and does not influence retrieval ranking.

**Rule 16.** Validity-time fields (`t_valid_start`, `t_valid_end`) are never fabricated. They are either set from explicit temporal claims in source text or left NULL.

These rules exist because each has a known failure mode they prevent: ungrounded extractions (mem0 audit), echo inflation (same), and confabulated timelines (general LLM hallucination tendency when asked for dates).

## 17. How to use this document — revised

Read order:

1. **v0.0** — full blueprint
2. **v0.1** — six contradiction fixes
3. **v0.2** — evaluation, cost, compaction, honest defaults, roadmap realism
4. **v0.3** — identity layer for digital twins
5. **v0.4 (this document)** — anti-hallucination hardening
6. **ADAPTER_WHATSAPP_MCP.md** — channel adapter example

v0.4 is the most defensive revision. It does not add capability; it prevents failures. The changes are motivated by evidence from other teams' production deployments rather than by theoretical concerns, and each one maps to a specific observed failure mode:

- Bi-temporal modeling → answers queries that pure-transactional systems can't answer correctly
- Grounding gate → stops mem0-style junk accumulation at the source
- Distinct-source counting → prevents echo-based confidence inflation
- Targeted contradiction detection → reduces false-positive supersessions that themselves cause hallucinations
- Identity-aware extraction → stops demographic fabrication for non-human subjects
- As-of queries → makes point-in-time retrieval honest rather than approximated
- Recalibrated eval targets → keeps us honest about what memory systems can currently achieve

If a v0.5 is produced it should target one of: multi-modal ingest (voice, images, structured data fusion), multi-persona coordination, or a formal threat model document. Those are real gaps that v0.1–v0.4 do not cover. They are not blockers for building.

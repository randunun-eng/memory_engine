# 0005 — Ed25519 for MCP signatures

## Status

Accepted — 2026-04-16

## Context

Every inbound event carries a signature that the engine verifies before writing to the log (CLAUDE.md §11 R1). The signature binds the content to an MCP's identity, making unsigned or tampered events rejectable at ingress.

The choice of signature algorithm affects:
- Verification latency per event (matters at high ingest rates).
- Key size (affects DB storage and wire size).
- Library maturity and ubiquity (MCPs may be written in any language).
- Quantum resistance (long-term consideration).
- Signature size (stored per event, forever).

Candidates:
- **RSA (2048+ bit).** Mature, universal. Large signatures (256 bytes), slower verification.
- **ECDSA (P-256, P-384).** Common in TLS. Standards-compliant (FIPS 186). Smaller signatures (~64 bytes). Historically had subtle implementation pitfalls (nonce reuse); modern libraries handle this.
- **Ed25519.** Based on EdDSA. Fast (both sign and verify), small (64-byte signature, 32-byte public key), fewer foot-guns than ECDSA. Widely supported: OpenSSH, libsodium, NaCl, ed25519-dalek, nacl.signing.
- **BLS signatures, Schnorr, various PQ.** Rejected as too niche or too experimental for a reference implementation in 2026.

For memory_engine specifically:
- Multiple MCPs per deployment over time; we want key generation to be trivial for new MCPs.
- Signatures are stored in every event row, forever. Smaller is better.
- Verification happens on every ingest. Fast is better.
- Sigs need to be producible in Python (engine side) and, for test MCPs, from any language.
- Quantum resistance is not a Phase 0–7 concern; events signed today that remain sensitive in 15+ years are rare in practice. We can revisit if needed.

## Decision

**Ed25519 via `pynacl` (the Python binding to libsodium).**

- Keys: 32-byte public, 64-byte private (seed form).
- Signatures: 64 bytes, base64-encoded to 88 chars in storage.
- Canonical signing message: `f"{persona_id}:{content_hash}".encode("utf-8")`.
- Library: `pynacl` in engine, `libsodium` or equivalent in MCPs.

Key generation handled by `memory_engine.policy.signing.generate_keypair()`. Private key is shown once to the operator at MCP registration; the engine never stores it.

Key rotation: `mcp_sources` table supports multiple rows per persona; old keys have `revoked_at` set when retired, but remain available for verifying historical events recorded before revocation.

## Consequences

**Easier:**
- Fast verification. ~100k signatures/sec on a typical CPU (Ed25519 is designed for speed).
- Small storage. 88 base64 chars per event signature.
- Fewer implementation footguns than ECDSA. No nonce-reuse concerns.
- libsodium bindings exist in every major language. MCPs can be written in Python, Node, Go, Rust, Java, etc., and all produce compatible signatures.
- pynacl is a thin wrapper around libsodium; audit surface is small.

**Harder:**
- Not FIPS 186 compliant. If a future deployment has strict FIPS requirements, we'd need ECDSA as an alternative. Low probability for Phase 0–7.
- Ed25519 is not quantum-resistant. In the post-quantum era, we'd need to migrate to a lattice-based scheme (Dilithium, etc.). The plan: when NIST finalizes PQ signature standards and mature libraries exist, rotate all MCP keys to the new scheme. Old signatures remain verifiable by keeping verification code for Ed25519.
- No HSM support out-of-the-box in pynacl. If an operator wants to store MCP private keys in an HSM (YubiHSM, AWS CloudHSM), they'd need custom glue. Phase 0–7 does not support this; documented as a gap.

**Future constraints:**
- Canonical signing message format is a contract. Changing it invalidates all historical signatures. If we need to change (e.g., to add a timestamp field to prevent cross-persona replay), it's a migration with operator notification and a planned transition window.
- Signature algorithm is stored per `mcp_sources` row (`public_key_ed25519` column name hints at the algorithm). Future-proofing: migration 007 (post-Phase-7) can add a `sig_algo` column if we support multiple algorithms.

## Alternatives considered

- **ECDSA P-256.** Rejected because Ed25519 is faster, smaller, and has fewer pitfalls.
- **RSA 4096.** Rejected for size (512-byte signatures are 8x Ed25519) and slower verification.
- **HMAC-SHA256 with a shared secret.** Rejected because shared-secret schemes conflate the roles of signer and verifier. MCPs sign; the engine verifies. Asymmetric crypto enforces this separation.
- **JWT.** Rejected because JWT's library ecosystem has a long history of implementation bugs (algorithm confusion attacks, none-algorithm acceptance). Rolling our own with pynacl is safer.
- **PQ signatures (Dilithium, Falcon, SPHINCS+).** Rejected for 2026 because library maturity is not yet at the bar we need for a reference implementation. Will reconsider in a future ADR when NIST standards are final and at least two independent implementations have been audited.

## Revisit if

- A FIPS-compliant deployment is required (→ add ECDSA as alternative, store algo per MCP).
- HSM-backed key storage becomes a documented requirement.
- NIST-standardized PQ signatures reach library maturity and key rotation becomes tractable.
- A cryptographic weakness in Ed25519 is discovered (no such thing as of early 2026; included for completeness).

| 2026-04-17 | env-vars | `env-var-delimiter-pydantic-settings`: `docker-compose.yml` originally used `MEMORY_ENGINE_DB_PATH` (single underscore between `DB` and `PATH`). pydantic-settings uses `__` (double underscore) as the nested-field delimiter, so the single-underscore form was silently ignored, causing memory-engine to write to the ephemeral `/app/src/data/engine.db` instead of the host-mounted `/app/data/engine.db`. `docker compose down` nuked all bootstrap state. | 2026-04-18 | twin-agent | `twin-agent-assumed-integer-message-id`: Bundle spec declared `IncomingMessage.id: int` and tracked checkpoint as integer, with SQL `WHERE m.id > ?` for monotonic ordering. Bridge's actual schema has `messages.id TEXT` (whatsmeow IDs like `3EB0A4C2B8F1234567`). At runtime, `int(CHECKPOINT_FILE.read_text())` raises `ValueError` on the first real TEXT id and silently resets checkpoint to 0. Left unfixed this would reprocess ~50 messages per 3-second poll = ~1000 LLM calls/min, blowing Gemini free tier's 15-RPM limit in under a minute with silent 429 failures and no visible draft output. **Propagated to control-plane**: `DraftIn.incoming_msg_id int→str`, `drafts.incoming_msg_id` column `INTEGER NOT NULL → TEXT NOT NULL`. Both changes required because message IDs are TEXT throughout lharries' schema. First symptom of the downstream bug was twin-agent drafting successfully but control-plane rejecting `POST /internal/drafts` with 422 `int_parsing` errors — silent loss of every draft. | Patched in twin-agent: `IncomingMessage.id: str`, checkpoint file renamed `last_processed_id → last_processed_ts` and stores ISO timestamp, `read_checkpoint() → str` defaulting to `SYNC_FROM_ISO`, `fetch_new_messages(since_ts_iso: str)` signature simplified, SQL filters by `timestamp > ?` and orders by `timestamp ASC`, main loop writes `msg.timestamp.isoformat()` after each processed message. All `log.info("...msg=%d..." % msg.id)` call sites updated `%d → %s`. Propagated in control-plane: `DraftIn.incoming_msg_id` is `str`, `drafts.incoming_msg_id` column is `TEXT NOT NULL`, duplicate-draft log line uses `%s`. The `drafts` table was empty at migration time (only failed-422 attempts had tried to write), so the cheap path was taken: delete the SQLite file + host-side `-shm`/`-wal` and let `init_db()` recreate with the new schema on container restart. `UNIQUE (incoming_msg_id, persona_slug)` index continues to work on TEXT columns. Remove DRIFT when canonical bridge schema is documented and both services have unit tests covering the TEXT-id contract. | Open |

| 2026-04-18 | twin-agent | `twin-agent-broadcast-messages-skipped`: WhatsApp Status updates use JIDs ending in `@broadcast` (e.g. `status@broadcast`). They are one-way broadcast content, not conversational messages, and generating draft replies to them is wasteful and nonsensical (`Draft for msg=...: 'Okay.'` in response to a TikTok link shared by a contact's Status). | `process_message` in twin-agent checks `msg.chat_jid.endswith("@broadcast")` alongside `is_group` and skips before ingest/recall/LLM. Checkpoint advances on the skip so broadcast messages aren't re-fetched. Same treatment for groups and empty-content messages, also explicit (defensive — outer main_loop writes the checkpoint too after a clean return, so this is idempotent). Alpha scope: individual conversational messages only. When group support is added post-alpha, the broadcast skip stays. | Open |
| 2026-04-18 | twin-agent | `twin-agent-timestamp-separator-mismatch`: Bridge SQLite stores `messages.timestamp` with a **space** separator (`2026-04-18 06:08:00+00:00`), but Python's `datetime.isoformat()` defaults to the ISO-T separator (`2026-04-18T06:08:00+00:00`). SQLite compares these as strings lexicographically — ' ' (0x20) sorts before 'T' (0x54), so every real message looks **older** than the ISO-formatted checkpoint and the `WHERE timestamp > ?` filter never matches. Silent failure: twin-agent runs cleanly, polls every 3s, and finds 0 messages forever while the bridge accumulates them. Verified on live DB: T separator → 0 matches; space separator → 46 matches. | `fetch_new_messages` passes `since_ts.isoformat(sep=" ")` to the query instead of default isoformat. Checkpoint file content stays in ISO-T format (it's read by Python, not compared in SQL), but all SQLite query params for `timestamp` comparison must use space separator. If the bridge ever changes its storage format to ISO-T, update this. Remove DRIFT when canonical bridge schema is documented and a smoke test asserts that `fetch_new_messages` returns a non-empty result when the bridge has inbound messages. | Open |
